# 项目审查规则

## 项目名称

政务公开触摸查询系统

## 核心架构文档

审查时必须读取并以以下文档为基线：

- `docs/database.md`
- `docs/architecture.md`
- `docs/api-spec.md`
- `CLAUDE.md`

同时读取：

- `docs/dev-logs/` 下最新一份交付报告
- 本次实际代码改动
- 本次构建、类型检查和测试结果

当交付报告与实际代码不一致时，以实际代码和验证结果为准，并在审查报告中指出差异。

## 审查目标

检查最新开发内容是否符合项目设计、任务范围和工程约束，判断本次交付是否可以验收。

审查不得只检查能否编译，还应检查：

- 核心业务是否可运行
- 原有功能是否发生回归
- 接口调用链路是否完整
- 交付报告是否如实反映实现情况
- 是否存在未声明的风险或遗留问题

## 必查项

### 架构

- 是否符合 `architecture.md`
- 是否遵循 `kiosk-app`、`admin-web`、`backend` 三工程结构
- 是否新增未规划模块
- 是否提前实现后续扩展模块
- 是否破坏模块职责或依赖边界
- 是否将管理端、群众端和外部平台职责混合
- 是否保留合理的扩展能力，避免无必要的过度设计

### 数据库

- 是否符合 `database.md`
- 是否使用 `varchar(36)` 主键和后端 UUID
- 是否使用自增主键
- 是否使用数据库 `ENUM`
- 是否使用数据库专属 JSON 类型
- JSON 配置是否按文本存储
- 是否包含规定的公共时间字段
- 是否使用逻辑删除
- 是否使用 MySQL 专属函数、触发器或存储过程
- 是否破坏 MySQL 8 与 HighGo 的兼容性
- 表、字段和关系是否与数据库基线一致

### API

- 是否符合 `api-spec.md`
- 管理端是否统一使用 `/api/admin/*`
- 群众端是否统一使用 `/api/public/*`
- admin/public 接口是否混用
- 管理端接口是否执行登录和权限校验
- 群众端接口是否只返回已发布内容
- 是否暴露后台字段、共享平台参数或凭据
- 是否使用统一响应结构
- 分页结构和错误码是否符合规范
- 写操作是否具备操作日志能力
- 发布操作是否记录发布记录
- 外部接口是否具备超时、异常处理和缓存兜底

### 群众端

必须满足：

- 不允许键盘输入
- 不允许输入框搜索
- 不允许出现会唤起系统键盘的可编辑控件
- 所有查询和导航均可通过点击或触摸完成
- 不允许直接调用大数据共享平台接口
- 不允许调用 `/api/admin/*`
- 顶部约 30% 主要用于展示
- 主操作区域在屏幕中下部
- 底部导航固定并包含首页、返回、重来、帮助
- 90 秒无操作自动返回首页
- 返回首页或重来时清除临时状态
- 断网或接口失败时显示友好提示，不允许白屏
- 适配 1080 × 1920 竖屏和大尺寸触控

### 管理端

- 是否使用规划的 Vue3、TypeScript、Vite、Element Plus、Pinia 技术栈
- 管理功能是否通过 `/api/admin/*` 调用后端
- 是否具备登录、权限控制和会话失效处理
- 内容编辑、审核和发布权限是否合理分离
- 是否避免在前端保存敏感凭据

### 审核发布

所有可发布内容必须：

- 支持状态流转
- 支持版本管理
- 已发布版本不得被直接覆盖
- 支持提交审核、通过、驳回、直接发布、撤回和回滚
- 直接发布必须受权限控制
- 公共接口不得返回草稿、待审核、驳回或已撤回内容
- 每次发布操作必须写入发布记录
- 管理端写操作必须兼容操作日志

### 办事指南

- 群众端是否只调用本系统 Public API
- 后端是否通过 `ServiceGuideModule` 调用共享平台
- 是否避免全量同步和定时全量同步
- 是否只在本地维护轻量展示配置
- 是否将共享平台返回转换为稳定的公开 DTO
- 是否实现超时控制和最后一次有效缓存
- 是否避免记录敏感个人信息

### 安全与隐私

- 是否泄露账号、密码、Token、密钥或平台凭据
- 日志是否记录身份证、手机号、人脸等敏感个人信息
- 上传接口是否限制文件类型和大小
- 是否存在未鉴权的管理端写接口
- 是否存在过度开放的 CORS、调试接口或错误堆栈泄露
- 是否对请求参数进行 DTO 校验

### 一期范围

禁止提前实现：

- AI 问答
- 智能预约
- 个人办件查询
- 群众身份认证
- 数字人
- 设备管理
- 打印
- 扫码

只允许为后续扩展保留清晰接口或模块边界，不得提前引入复杂业务实现。

### 工程质量

- 是否只修改了本次任务直接相关的文件
- 是否存在无关重构或依赖膨胀
- 是否保留锁文件并保证依赖可复现
- 是否通过可用的类型检查、构建和测试
- 是否包含必要的接口冒烟测试
- 是否出现原有功能回归
- 部署配置是否与当前目录和构建产物一致
- 交付报告是否列明未完成事项和风险

## 审查结果

审查结果只能使用以下三种结论：

### 通过

实现符合设计文档和任务范围，没有阻断使用或验收的问题，验证结果完整。

### 有条件通过

核心目标已经完成，没有阻断性问题，但存在不影响当前使用的轻微缺陷或明确可延期事项。

### 不通过

存在以下任一情况：

- 核心功能不可用
- 出现明显功能回归
- 违反系统架构、数据库兼容或接口隔离规则
- 违反群众端零键盘等强制约束
- 存在高风险安全问题
- 提前实现被禁止的一期外功能
- 交付内容无法构建、运行或验证

## 输出内容

审查报告必须包含：

1. 审查结论
2. 符合预期的部分
3. 不符合预期的部分
4. 风险点
5. 需要修复的内容
6. 下一步 Claude Code 提示词

问题应尽量标明具体文件和行号，并按严重程度排序。

下一步提示词必须：

- 可以直接复制给 Claude Code 使用
- 明确要求读取核心基线文档和最新交付报告
- 明确允许和禁止修改的范围
- 列出必须完成的验证命令
- 要求生成新的 `docs/dev-logs/` 交付报告
- 不得用“构建通过”替代核心业务链路验证
